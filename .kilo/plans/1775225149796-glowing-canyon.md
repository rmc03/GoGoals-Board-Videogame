# GoGoals - Code Review & Improvement Suggestions

## Overview

GoGoals is a well-structured educational board game built in Godot 4.5. The architecture is clean, documentation is excellent, and the signal-driven communication pattern is well-implemented. Below are the findings organized by priority.

---

## CRITICAL Issues

### 1. GameManager is a God Class (464 lines)
**Files:** `scripts/Managers/GameManager.gd`

`GameManager` handles turn orchestration, dice, movement animation, quiz flow, audio, pause/resume, victory detection, and match statistics. This is the primary complexity bottleneck.

**Recommendation:** Extract into focused service classes:
- `TurnManager.gd` — turn rotation, player index tracking
- `MovementService.gd` — path calculation, movement animation, tile resolution
- `QuizService.gd` — question fetching, quiz state, answer resolution
- `MatchStatsTracker.gd` — statistics collection and summary

`GameManager` becomes a thin orchestrator that delegates to these services.

### 2. Duplicate TileType Enum
**Files:** `scripts/Core/Constants.gd:61-68`, `scripts/Entities/Tile.gd:10-17`

Two separate `TileType` enums exist with slightly different values (`SPECIAL_LADDER`/`SPECIAL_SLIDE` vs `LADDER`/`SLIDE`). This is a maintenance hazard and source of potential bugs.

**Recommendation:** Keep only `TileEntity.TileType` in `Tile.gd`. Remove the duplicate from `Constants.gd`.

### 3. GameState Has Redundant Methods
**Files:** `scripts/Core/GameState.gd:43-44`, `scripts/Core/GameState.gd:71-72`

`get_current_player()` and `active_player_idx()` do the exact same thing. `increment_turn_count()` and `advance_turn()` overlap conceptually with the manager's turn logic.

**Recommendation:** Remove `get_current_player()` (use `active_player_idx()`). Consolidate turn logic so `GameState` is purely a data holder and `GameManager` owns turn policy.

---

## HIGH Priority

### 4. All UI Built Programmatically
**Files:** All UI scripts (`GameHUD.gd`, `QuizPanel.gd`, `PauseMenu.gd`, `MenuPrincipal.gd`, `EndGameMenu.gd`, `ComoJugar.gd`, `OptionsMenu.gd`)

Every UI element is created via code with `Control.new()`, `Label.new()`, `Button.new()`, etc. This makes visual iteration extremely difficult — changing spacing, colors, or layout requires code changes and re-runs.

**Recommendation:** Convert at least the static screens (MenuPrincipal, ComoJugar, EndGameMenu) to use `.tscn` scene files with Godot's visual editor. Keep dynamic HUD elements programmatic if needed.

### 5. Hardcoded Magic Numbers
**Files:** Multiple

Examples:
- `GameState.gd:79` — `return 62` instead of using board size
- `GameHUD.gd:378` — `range(8)` for dice animation frames
- `QuizPanel.gd:359` — `get_tree().create_timer(1.8)` hardcoded feedback duration
- `MenuPrincipal.gd:97` — `DEBUG_SEED_RANKING` constant
- All UI position/size values scattered across scripts

**Recommendation:** Move tunable values to `Constants.gd` or export variables. The `GameState.get_finish_tile_index()` should query the board entity or use a constant shared with `BoardConfig`.

### 6. No Input Validation for Player Name
**File:** `ui/EndGameMenu.gd:301-308`

The validation only checks if the name is empty. No length limit, no sanitization.

**Recommendation:** Add max length (e.g., 20 chars), trim whitespace, and potentially filter special characters.

### 7. RecordsManager Name Comparison Bug
**File:** `autoloads/RecordsManager.gd:45-53`

The `submit_new_record` function iterates and removes during iteration with `records.remove_at(i)`, then appends. This can cause issues if the same name appears multiple times and the logic is fragile.

**Recommendation:** Use `filter()` to find existing records, then replace or append cleanly.

## MEDIUM Priority

### 8. No Accessibility Features
The game has no:
- Colorblind mode (quiz feedback relies on green/red)
- Text scaling options
- Screen reader support hints
- Alternative visual indicators beyond color

**Recommendation:** Add icons alongside color feedback (checkmark/X), and consider a colorblind-friendly palette option.

### 9. Audio Stream Dependencies Not Validated
**File:** `scripts/Managers/GameManager.gd:101-118`

`configure_audio()` accepts `AudioStream` parameters but doesn't validate them. If any are null, the game silently skips audio without warning.

**Recommendation:** Add `push_warning()` for null streams in debug builds, or preload defaults.

### 10. Question History is Global, Not Per-Session
**File:** `autoloads/GameData.gd:6`

`question_history` persists across game sessions until `reset_question_history()` is called. If the game crashes or exits without calling it, history may carry over.

**Recommendation:** Reset history on each `initialize_game()` call (already done via `MainBoard`, but make it more robust by also resetting in `GameData._ready()` or on scene change).

### 11. `_apply_window_mode` Has Redundant Logic
**Files:** `scripts/UI/Game/PauseMenu.gd:168-181`, `scripts/UI/Menu/OptionsMenu.gd:239-252`

Both files contain nearly identical window mode handling code. The logic for setting fullscreen, then checking and switching to exclusive fullscreen, then setting maximized is contradictory — setting `mode` twice in sequence.

**Recommendation:** Extract to a shared utility class `WindowManager.gd` as an autoload. Fix the logic: choose ONE mode, don't set it multiple times.

### 12. No Loading Screen
Scene transitions (`change_scene_to_file`) are instant in the editor but may stutter on lower-end devices or exported builds.

**Recommendation:** Use `ResourceLoader.load_threaded_request()` and a loading screen for the game scene.

### 13. Player Offset is Hardcoded and Cumulative
**File:** `scripts/Entities/Player.gd:58`

`offset = Vector2(player_index * 10, player_index * 5)` — with 4 players, player 4 is offset by (30, 15), which may cause overlap issues on small boards.

**Recommendation:** Make offset configurable via `Constants.gd` and consider a circular arrangement around the tile center.

### 14. No Internationalization (i18n)
All text is hardcoded in Spanish. The game's educational nature (SDGs) would benefit from multi-language support.

**Recommendation:** Use Godot's built-in translation system (`tr()`) with CSV translation files.

## LOW Priority

### 15. `_setup_visual()` in TileEntity is Empty
**File:** `scripts/Entities/Tile.gd:31-45`

The method exists but only has `pass` in each match arm. Visual configuration is handled externally by `BoardTileVisual.gd`.

**Recommendation:** Remove the empty `_setup_visual()` method or document why it exists as a hook.

### 16. IconoFlotante Uses Deprecated Pattern
**File:** `scripts/IconoFlotante.gd`

Uses `extends Sprite2D` with manual `_process` sin/cos animation. Godot 4 has built-in animation tools.

**Recommendation:** Replace with an `AnimationPlayer` tween or Godot's built-in sine-based animation in the editor.

### 17. Test Suite is Minimal
**File:** `tests/run_tests.gd`

Only 4 tests covering basic functionality. No UI tests, no integration tests, no edge case coverage.

**Recommendation:** Add tests for:
- Victory detection with exact roll
- Quiz answer with correct/incorrect flows
- Ladder-to-ladder chaining
- Records sorting edge cases
- Pause during movement (should be blocked)

### 18. No Error Recovery for Missing Questions File
**File:** `autoloads/GameData.gd:12-28`

If `questions.json` is missing or malformed, the game starts with an empty question bank. Quiz tiles will silently skip.

**Recommendation:** Show a user-visible error or fallback. Consider bundling a minimal question set as a fallback.

### 19. Constants.gd Extends Node Unnecessarily
**File:** `scripts/Core/Constants.gd:1`

`extends Node` with `class_name Constants` — constants don't need to be nodes. They should be a plain `class_name` without extending anything, or a static class.

**Recommendation:** Change to `class_name Constants` without `extends Node`. Update any references that use it as a node path.

### 20. Unused Signals
**Files:** `BoardEntity` has `board_loaded` signal that is never connected. `PlayerEntity` has `position_changed` and `texture_changed` signals that are never used.

**Recommendation:** Remove unused signals or wire them up for debugging/telemetry.

---

## Architecture Strengths

- **Signal-driven communication** between UI and game logic is well-designed
- **GameState separation** makes state inspection and testing straightforward
- **Documentation quality** is excellent (ARCHITECTURE.md, HOW_THE_GAME_WORKS.md)
- **Test suite** exists and runs headless — good foundation to build on
- **Data-driven quiz content** with cycle-without-repeats logic
- **Match statistics tracking** is comprehensive

---

## Suggested Implementation Order

1. **Remove duplicate TileType enum** (quick fix, eliminates confusion)
2. **Consolidate GameState redundant methods** (quick cleanup)
3. **Extract MovementService from GameManager** (biggest impact on complexity)
4. **Extract QuizService from GameManager** (second biggest impact)
5. **Move magic numbers to Constants** (improves maintainability)
6. **Convert static UI scenes to .tscn** (enables visual editing)
7. **Add i18n support** (expands audience)
8. **Add accessibility features** (colorblind mode)
9. **Expand test coverage** (improves confidence in changes)
10. **Shared WindowManager autoload** (eliminates code duplication)
